Purpose: Allow operators to control what language the LLM uses in summaries, blocking inappropriate terms while allowing context-specific exceptions.
Output: language_filter.py module, default blocklist.txt, example allowlist.txt, tests, and integration into summarizer + startup.

    1. `load_terms(filepath: Path) -> list[str]`:

       - Split on newlines, strip whitespace, skip empty lines and lines starting with `#` (per Research pitfall 5)

       - If entry contains `(` and ends with `)`: split on first `(`, return (term.strip(), reason.strip()) (per D-04)
       - Otherwise return (entry.strip(), "")

    3. `build_language_guidelines(blocked: list[str], allowed: list[str]) -> str`:
       - If blocked is empty, return "" (no guidelines needed)
       - Build a string starting with `"\n\n## Language Guidelines\n"`
       - Add instruction: "When writing your summary, do NOT use any of the following terms or phrases. Also avoid obvious variations, misspellings, abbreviations, and leetspeak substitutions of these terms:" (per D-06)
       - List each blocked term as `"- {term}"`
       - If allowed is not empty, add "Exceptions" section with each entry parsed via parse_allowlist_entry, formatted as `"- \"{term}\" (acceptable because: {reason})"` if reason exists, else `"- \"{term}\""` (per D-04)
       - End with: "If a discussion topic involved blocked language, summarize the topic's substance without using the blocked terms themselves." (per Research pitfall 3)

    4. Module-level cache:
       - `_cached_guidelines: str = ""` module-level variable
       - `load_language_config(blocklist_path: Path | None = None, allowlist_path: Path | None = None) -> None`:

         - Log info with rough token estimate: `len(guidelines) // 4` tokens (per Research open question 2)
